# Agent Safety Constraints Specification - Phase 7A

This document establishes the safety constraints, execution sandboxes, and verification protocols that govern all agents in `bbc_aos`.

---

## 1. Method Allowlist Enforcement

Agents are restricted to an explicit method allowlist. The **BBC Orchestrator** rejects any message calling a method not on this list.

### Allowed RPC Methods by Agent

* **`PlannerAgent`:**
  * `create_plan`: Generate steps and checkpoints from a user request.
* **`ContextAgent`:**
  * `get_symbol_context`: Retrieve code structures for specific symbols.
  * `get_file_context`: Read target file code.
* **`ResolverAgent`:**
  * `resolve_dependencies`: Compile list of dependencies for a target file.
  * `calculate_blast_radius`: Calculate impact score for proposed changes.
* **`CompressionAgent`:**
  * `optimize_context`: Optimize prompt inputs using token minimization.
  * `semantic_pack`: Pack context structures.
* **`VerificationAgent`:**
  * `check_hallucinations`: Verify that a patch does not use unknown symbols.
  * `verify_syntax`: Check that the patched code compiles.
* **`DocumentationAgent`:**
  * `generate_notes`: Create development wikis or Obsidian note contents.
* **`ExecutionAgent`:**
  * `generate_patch`: Construct file patch diffs.
  * `apply_patch`: Write verified code changes to disk.

*Note: Any method call containing wildcard characters or unauthorized actions triggers a critical security event.*

---

## 2. Filesystem & Environment Sandboxing

Agents do not have raw shell or execution environments. They are sandboxed according to the following constraints:
* **Workspace Path Enforcement:** The `ExecutionAgent` and `ContextAgent` are restricted to files within the active project workspace root (`C:\Users\90535\.gemini\antigravity\scratch\BBC_AOS_Wiki` or similar). Absolute system directories (e.g. `/Windows`, `/Program Files`) are forbidden.
* **System Command Restrictions:** Agents are prohibited from spawning child processes (`os.system`, `subprocess.Popen`) or performing shell executions.
* **Network Isolation:** Agents run in network-isolated environments. No external API queries or remote sockets are permitted within the Agent Layer. All LLM communications are handled through the central orchestrator's gateway.

---

## 3. LLM Output Sanitization & CVP Triggers

All outputs generated by external LLM services are treated as untrusted. They must pass through the **Constraint Violation Protocol (CVP)** before being applied:

```
[Raw LLM Output] ──> [Schema Check] ──> [AST Syntax Check] ──> [Hallucination Check] ──> [Commit]
```

1. **Schema Check:** The orchestrator verifies that the LLM response is valid JSON matching the target template. If malformed, a `-32600` RPC error is raised.
2. **AST Syntax Check:** The `VerificationAgent` parses the code with `ast.parse()`. If syntax is invalid, a `SYNTAX_ERROR` is logged, and the patch is rejected.
3. **Hallucination Check:** The proposed patch is scanned against the symbol graph. If it references functions or classes not present in the index, a `HALLUCINATION_DETECTED` warning is generated, triggering the CVP block.
4. **Degenerate Scalar State:** If any numerical computation hits a `DEGENERATE` state during validation, it triggers a `MATH_CORE_DEGENERATION_ANOMALY` security warning, and execution is aborted immediately.
